内核设计文档 · 写给完全没接触过 ANFS 的人

ANFS Agent-Native File System —— 一个把"文件操作"变成"可溯源、可回放、省 token 的语义事实"的文件系统内核。

这份文档假设你从未听说过 ANFS。读完之后，你会理解：它到底是什么、为什么不能用普通文件系统、它的每一块是怎么设计的，以及它和传统文件系统在最底层有什么本质区别。文中所有代码都直接引用自仓库真实源码（标注了文件与行号），不是伪代码。

一句话
传统文件系统给 agent 的只有 path → bytes。ANFS 给 agent 的是 bytes + 谁在何时为何改了它 + 它从哪些输入派生而来 + 它能不能被某个 agent 看到 + 任意历史时刻它长什么样——而且这一切对模型暴露成它最擅长的"文件 + 目录 + grep"接口。

01为什么传统文件系统对 Agent 不够用

普通文件系统（ext4、APFS、NTFS……）是为人和长生命周期进程设计的。它的世界观是：一个路径指向一块可以随时被原地覆盖的字节。这套模型用了五十年，但当使用者换成大量并发、需要被审计、按 token 计费的 AI agent 时，四个缺陷立刻致命：

Agent 真正要问的问题传统文件系统的回答后果
这个文件为什么是现在这样？谁、哪一步、基于什么改的？无法回答。覆盖写之后旧内容和原因永久消失。无法审计、无法复盘 agent 的决策链。
这个输出是从哪些输入派生的？引用了哪些证据？没有"派生"概念。文件之间没有因果边。无法做血缘追踪、无法验证"答案是否真有依据"。
能不能回到三步之前那一刻的完整目录视图？需要外挂快照/git，且是粗粒度、非事件级。无法精确回放某个事件边界的状态。
这段敏感字节能不能让这个 agent 看到 / 写进输出？权限只到"文件能否打开"，不到字节、不到用途。无法做细粒度、面向数据流的策略控制。

你可以在普通文件旁边外挂一个 JSONL 审计日志、一个向量索引、一个 git 仓库来打补丁。但它们彼此不一致、不在同一个事务里、随时会和真实文件漂移。ANFS 的主张是：这些能力不该是外挂，而应该是文件系统内核本身的一等公民，和"写字节"在同一个事务里原子提交。

设计哲学
接口要 boring，底层要 agent-native。 上层永远是熟悉的 read / write / ls / grep / find / mkdir——因为模型对路径和文件最在行。底层却是事务化的语义内核：CAS 字节、不可变节点、可变引用、append-only 事件、血缘边、策略决策、可回放快照。

02心智模型转变：从"覆盖"到"事实 + 派生"

理解 ANFS 只需要先接受一个转变。传统文件系统是可变的：写就是覆盖，历史即丢失。ANFS 把世界分成两层：

┌─ CANONICAL 层（事实，永不可变 / 永不丢失）──────────────────┐ │ │ │ blobs 内容寻址的字节（按 sha256 去重） │ │ nodes 不可变的"某时刻某文件的内容快照" │ │ refs 可变的名字 → node 映射（唯一能"变"的东西） │ │ events append-only 操作日志（每个动作一条，带全序 seq） │ │ edges 事件 ↔ node 的输入/输出因果边（血缘） │ │ policy append-only 的标签 / 决策 │ └────────────────────────────────────────────────────────────┘ │ 纯函数投影，可丢弃、可重建 ▼ ┌─ DERIVED 层（派生索引，可删可重建，绑定 blob_hash 校验）──────┐ │ FTS 全文索引 · chunk 分块索引 · 向量 embedding · 物化工作集 │ └────────────────────────────────────────────────────────────┘

关键直觉："写一个文件"在 ANFS 里不是覆盖，而是产生一个新的不可变 node，然后把可变的 ref 指向它，同时记下一条事件和它的因果边。 旧 node 还在，旧 ref 指向也在 ref 历史里。于是"现在的样子"只是"全部历史事实"的一个最新切片——任何过去的切片都能精确重建。

03五个核心对象

整个内核只围绕五个名词运转。先建立这五个词的直觉，后面所有机制都是它们的组合。

字节层 · 不可变
Blob

内容寻址的原始字节。键就是内容的 sha256。两个相同内容的文件天然共享同一个 blob（去重）。

语义层 · 不可变
Node

一个"内容快照"：指向某个 blob，外加 kind、media_type、元数据、创建时间。一旦创建永不修改。

命名层 · 可变
Ref

名字 → node 的映射，系统里唯一会"变"的东西。如 ws:coder/src/app.py → node:abc，带 state（draft/published/approved…）。

历史层 · append-only
Event

每个操作一条不可变事实：write、read_ref、search、query、merge…… 带全局递增 seq，构成全序时间线。

因果层 · append-only
Edge

事件与 node 之间的 input/output 边。它把"这个输出由哪些输入派生"显式记下来，是血缘图的边。

用一行类比对齐你已有的直觉：

传统: path ───────────────────────► bytes （就这两个，覆盖即失忆） ANFS: ref ──► node ──► blob （命名/快照/字节 三段分离） │ ▲ │ │ output edge └──────────► event ◄──── input edge ──── 其他 node （谁/何时/为何 + 派生自谁）

04存储层：CAS 内容寻址（blob.rs）

最底层是 Content-Addressed Storage：字节的地址就是它自己内容的哈希。写入时按大小自动选择两种物理形态——小内容直接内联进 SQLite（省一次文件系统往返），大内容落成独立对象文件。阈值是 64 KiB：

为什么这样省 token / 省空间
内容寻址意味着相同内容只存一份。一个 agent 反复保存几乎一样的草稿、或多个 workspace 引用同一份只读资源时，blob 自动去重；node/ref 只是轻量指针。

05不可变性：用数据库触发器把"事实"焊死

"事实永不改写"不是靠程序员自觉，而是由 SQLite 触发器在内核层强制。任何对不可变表的 UPDATE / DELETE 都会被数据库直接 ABORT。例如 nodes 不可改、events 不可删：

仓库里有 26 张这样被触发器保护的 canonical 表。这条设计是后面一切"可溯源、可回放、可审计"的物理地基：如果事实能被偷偷改写，溯源就是假的。

06一次 write 到底发生了什么

这是理解 ANFS 的"啊哈"时刻。在传统文件系统里，write("a.py", bytes) 就是覆盖。在 ANFS 里，一次写在同一个事务内原子地产生四样东西：

  1. 新 node（指向去重后的 blob）——内容快照；
  2. 一条 write 事件——谁(agent)、哪次运行(run)、哪个工具调用、哪个 workspace；
  3. output edge——这次事件"产出"了新 node；
  4. 若干 input edge——若本次写派生自其他 node，记下"派生自谁"。

事件和边的写入函数本身很朴素——重点不在它们复杂，而在于它们和"写字节"在同一个事务里，要么全部成功，要么全部回滚。审计永远不会和数据漂移：

这就是与传统 FS 的分水岭
传统 write 之后，"上一版是什么、谁改的、为什么"全部消失。ANFS 的 write 之后，旧 node 仍在、写事件仍在、派生边仍在——覆盖被替换成了"追加一个新事实 + 移动一个指针"。

08血缘图：输出从哪些输入派生

因为每个事件都带 input/output 边，整个系统天然是一张因果图：node 经由事件连到其他 node。"血缘"就是在这张图上做递归遍历。下面是顺着 input 边向上游追溯祖先的真实 SQL（递归 CTE）：

这套机制最漂亮的应用是派生写：当 agent 基于若干来源产出一个新文件，调用 write(content, derived_from=[来源 node])——来源被记成 input 边（role source），新文件 node 是 output 边。于是事后可以精确回答"这个产物派生自哪些来源、来源又派生自哪些原始资料"——一条可验证的证据链，而不是模型的自说自话。（早期有专门的 answer API 做这件事，现已并入通用的 write + derived_from，让 kernel 回到最小通用模型。）

派生 node ◄──output── write 事件 ──input──► 来源 node A (source) ├──input──► 来源 node B (source) └─ 来源 node 各自又是更早 write/search 事件的 output … （沿 input 边递归下去 = 完整血缘）

09检索即上下文：把"模型看了什么"也变成事实

这是 ANFS 区别于"普通向量库 / RAG sidecar"的核心。当 agent 调用 search / query / context_pack，命中的 node 会作为 input 边写进事件，payload 里还记录 token 估算。换句话说，"模型在做这个决定时到底看了哪些字节、花了多少 token"本身就是一条可审计、可回放的事实。

省 token 的正确姿势
ANFS 的省 token 不是"压缩文本"，而是让 agent 先用结构化查询定位，再只取必要片段，并且把"取了什么"记账。普通 codegraph 也能省 token，但它丢了审计与血缘；ANFS 的省 token 动作本身可溯源、受策略约束——这是同时满足"省 token"和"可溯源"两条第一性原理的关键。

10派生索引层：可丢弃、可重建、绑哈希校验

FTS 全文索引、chunk 分块、向量 embedding、物化工作集——这些都是 DERIVED 层。它们的设计纪律是：是 canonical 字节的纯函数投影，可以随时删掉再重建，且每行都绑定 blob_hash 用于校验。例如 chunk 索引表把分块的 offset/size/sha256 与源 blob 的 hash、size 一起存，任何漂移都能被发现：

派生表粒度可重建校验字段
node_fts整 node 文本从 node 内容重建
node_chunk_index固定字节分块blob_hash, blob_size, sha256
node_embeddings整 node 向量model, norm（缺 model_version，已知技术债）
fragments / fragment_edges代码 symbol / 调用边（§13）blob_hash, parser, parser_version
materialized_working_sets整个命名空间快照manifest_hash

重建入口 rebuild_derived_indexes()（manifest.rs）会清空并重算这些索引——因为它们不是事实，丢了也能从 canonical 层恢复。这条"事实 / 派生"分层正是健壮性的来源：派生层损坏不影响数据真相，重建即可。

大文件：内容寻址分块存储
大文件可存成内容寻址 chunk blob + 一个小 manifest node（write_chunked / read_chunked_node）。相同 chunk 跨文件、跨编辑自动去重，小改动只重存受影响的 chunk。chunk blob 只被 manifest 引用，所以 GC 专门保护它们、integrity 的 check_chunked_nodes 校验它们都在——派生层省空间，canonical 不变。

11字节级权限：不止"能否打开文件"

传统权限只能回答"这个用户能否打开这个文件"。Agent 需要更细的问题："这一段敏感字节，能不能让这个 agent 看到、能不能写进输出？" ANFS 用 fragment policy label 给文件的任意字节范围打标签：

读取 / 搜索 / 回答 / 导出时，可见性层会检查这些 label：带禁用 label 的字节范围或整个 node 直接被屏蔽，连进不进上下文都由内核裁决：

策略分三个层级：ref 级（这个名字）、node 级（这块内容）、fragment 级（内容里的某段字节）。而且因为标签本身也是 append-only 事件，"谁在何时给哪段字节打了什么标签"同样可审计、可回放。

12工作区隔离与三方合并

多个 agent 并发工作时，各自有独立命名空间（如 ws:agent-7/...），写操作是写时复制（copy-on-write）的 ref 更新，互不干扰。要把一个 workspace 的改动并回共享主干时，ANFS 做的是 git 式三方对比：比较 ① checkout 时的基线快照、② 主干当前状态、③ workspace 状态——只有"双方都改了且改得不一样"才算冲突，且冲突在动共享内存之前就被检出：

非冲突的改动可以原子合并；冲突被明确报出而不是悄悄覆盖。整个 event_sequence 提供全序，让并发也有确定的审计与回放。

13代码图：原生理解结构、省 token

这是把"agent 省 token"做成 ANFS 一等能力的一层——不是外挂 codegraph，而是把代码当作 node 字节的派生投影：用 parser 把文件解析成 fragment（命名的字节范围 = symbol）和 edge（calls / imports），存进通用表，绑 (node_id, blob_hash, parser, parser_version) 保证可重建。加一门语言 = 加一个 parser，别处不动（已接 Markdown / JSON / tree-sitter Rust）。

实测：对本仓库 worktree.rs 打包 commit_worktree 的上下文——整文件约 10499 token → 1474 token，省 86%。而且 context_pack 带 agent 身份时会写一条 code_context_query 事件（input 边指向打包的 node），并用 §11 的字节级 label 过滤掉敏感片段——省 token 的动作本身可审计、受策略约束，这是相对普通 codegraph 的差异点。

候选集即多条边，不存概率
跨文件调用 foo() 可能对应多个同名定义。ANFS 不存 confidence 分数——确定性系统不存猜测；一个名字对应多个目标 = 同证据下的多条 edge（与"一个事件多条 input 边"同构），查询时如实给出完整候选集。跨文件 resolve 只在 workspace 组合层做，绝不写回按 blob_hash 的派生表。

14Replay：回放到任意事件边界

既然每个 ref 变更都带 seq，"第 N 个事件那一刻的目录视图"就是一次确定性查询：取每个 ref 在 seq ≤ N 范围内的最后一次变更（用窗口函数 ROW_NUMBER 取 rank=1），即得当时的完整视图。这就是事件级时间旅行，比文件系统快照精细得多：

为什么传统 FS 做不到
普通文件系统覆盖写后旧字节已不存在；要"回到过去"只能靠外挂的整盘快照或 git，且粒度是提交/快照点，不是"第 N 次操作"。ANFS 因为从不丢弃事实，任意事件边界都能用一条查询重建——这是不可变 + 全序 seq 的直接红利。

15Worktree：与真实工作目录双向同步

agent 有时需要把内容物化成真实磁盘目录（比如让一个真实的编码工具/编译器去操作），再把改动提交回内核。这就是 worktree（src/worktree.rs）：把 workspace 的 ref 视图导出成普通目录，外部进程随便改，之后扫描目录把变更作为新一批 node + 事件提交回来。

为了保持内核的纯净，这个适配器是保守的：拒绝符号链接、特殊文件、含反斜杠的相对路径，以及保留的 .anfs-worktree-manifest.json 路径——任何不能被干净映射回 canonical 模型的东西都被挡在门外，而不是污染事实层。它既兼容真实工具链，又守住了"边界清晰"。

16完整性自检：把不变量变成可执行检查

所有这些不变量（哈希匹配、事件序号连续、ref 指向存在的 node、派生索引与源一致、策略边引用存在……）都不是口头承诺，而是 verify_integrity() 里27 项可执行检查。这是"健壮"这条第一性原理的兜底：

可溯源

代码图已落地（见 §13），真正的后续

"用派生投影把代码变成可省 token 的结构图"已经实现并合并进 main：统一 fragment/edge 表、tree-sitter Rust、context_pack 实测省 86% token、code_context_query 审计事件。还在路上的是：

  • 内容定义分块（CDC）：当前大文件分块是定长，对"文件开头插入"不鲁棒；CDC 让插入也能复用未变 chunk；
  • 更多语言 parser + 按 media_type 自动选 parser（目前由调用方显式选）；
  • workspace 隔离：在"一次性 session 工作区"模型下，owner 绑定被有意搁置（没有别的 agent 需要拦，是伪问题）；fork_workspace 已作为死概念移除，kernel 回到最小通用模型。

一句话：普通 codegraph 只省 token；ANFS-native 代码图在省 token 的同时保留审计、血缘与权限——这已经是 main 上的现实，不再是设想。

19术语表

CASContent-Addressed Storage，内容寻址存储，字节的地址即其哈希。
Blob不可变的原始字节，按 sha256 去重。
Node不可变的内容快照，指向一个 blob。
Ref可变的名字→node 映射，系统中唯一会变的东西。
Eventappend-only 的操作事实，带全局递增 seq。
Edge事件↔node 的 input/output 因果边，构成血缘图。
Canonical 层不可变事实（blob/node/ref/event/edge/policy）。
Derived 层派生索引（FTS/chunk/embedding），可丢可重建。
Fragment label打在文件某字节范围上的策略标签。
Workspace每个 agent 独立的命名空间，写时复制隔离。
Replay用 seq 重建任意历史事件边界的目录视图。
Worktree把 ref 视图物化成真实磁盘目录并可提交回内核的适配器。
