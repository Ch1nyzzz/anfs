Agent-Native File System —— 一个把"agent 真正需要、而原生文件系统留给应用层"的保证，下沉为内核不变量的语义文件系统。
论点（机制 vs 劝说）：当下 agent 的保证做在提示词层（"别泄漏 PII"）、应用层（自己加锁、手写日志）或干脆缺失——这些是概率性、会失效的。ANFS 把它们做成文件系统内核里可验证的结构性保证，同时不破坏 agent 已经训练过的 file/path 接口。下面四项证明，每一项都配一个会当场翻车的对照组。

什么样的文件系统才算 "Agent-Native"？

原生文件系统（POSIX / ext4 / 对象存储）给你的是字节和路径。它不理解：数据带不带敏感标签、有多少 agent 在并发改同一份状态、产物是从哪些来源派生来的、怎么把某个时刻的工作集重建出来。这些都被甩给应用层去"自求多福"。

"Agent-native" 的定义因此是两件事的并集：(1) 保留 agent 已经训练过的 file/path 操作，(2) 把 agent 需要、原生留给应用的那些保证，做进内核。下面 C1–C3 是原生没有的差异化保证，C4 证明这些保证不以兼容性为代价。四者合起来，就是"agent-native"被操作化、被实测的样子。

四项证明 · 每项配一个会翻车的对照

证明力不来自"我们造了功能"，而来自同一场景下对照组当场出错、ANFS 不出错。所有数字来自 benchmarks/ 下可复现的脚本，并由 tests/ 锁成回归断言。

C1 · 字段级策略 · 防泄漏

敏感数据在内核被拦住，与提示词无关

同一个真实模型（deepseek-v4-flash）、同一条"绝不泄漏 PII"系统提示、同一批攻击，只有上下文组装方式不同。攻击是无害的重格式化任务（"把这条记录整理成 JSON"）——这才是真实泄漏向量：对齐模型会拒绝"告诉我 SSN"，却会乖乖复述它没意识到敏感的字段。

prompt 防御
75% 泄漏
正则 scrubber
25% 泄漏
ANFS 内核
0% 泄漏

prompt 防御漏掉全部字段（SSN/邮箱/工资/诊断/病历号/API key/电话）；scrubber 抓住格式化 PII 但漏掉语义字段（工资 185000、"hypertension stage 1"、MRN-558831）；ANFS 字段从未进入上下文。

为何是 agent-native：agent 的核心动作是"检索文档进上下文再生成"。原生 FS 只有文件级 ACL，挡不住"字段"。ANFS 让字段级标签成为内核 gate（visibility.rs），无论提示词如何、模型是否认出敏感性，受标字节都不过 API。harness：leak_llm_baseline.py · leak_redteam_benchmark.py
C2 · 并发协调

文件系统出面协调并发写，而不是甩给应用层

4 个真 OS 进程开同一个 ANFS db，barrier 同时释放去抢合并。三向对照里 git 不是稻草人——它是带合并协议的应用层协调。

场景plain 目录gitANFS
整文件并发改静默 last-writer-wins报冲突报冲突
同文件异行改静默 last-writer-wins静默自动合并（无人审）浮为冲突待审

真多进程 contention：恰 1 个胜出、3 个拿到干净 RefConflictError、0 静默丢失、败方数据可取回、integrity 干净；disjoint：4 个全部干净合并、0 误报。

为何是 agent-native：文章的核心场景就是"数百 agent 并发改共享状态"。原生共享目录是 last-writer-wins，agent 的工作互相静默覆盖。ANFS 在内核每次写都做乐观版本检查，竞争一律浮出来。这个 harness 还抓到并修了一个真内核 bug：merge 路径用 DEFERRED 事务，真并发下漏出裸 SQLITE_BUSY；改用 TransactionBehavior::Immediate 后归零——模拟版永远发现不了。harness：concurrency_real_benchmark.py
C3 · 溯源 / Replay

任意产物的派生谱系可重建、任意时刻可回放

建一条 agent 派生链：导入 3 个源 → 从其中两个派生出 brief → 从 brief + 第三个源派生出 report。问：给定最终产物，能追溯出哪些上游源？

ANFS 派生闭包
4/4 · 100%
git 结构恢复
0/4 · 0%

git --follow 只给出 report.md 自己的 commit 历史——它结构上没有"派生自"这个关系；ANFS 把派生记成 typed 边，重建出完整闭包，并能 replay 到任意事件边界（比 commit 粒度更细）。integrity 干净。

为何是 agent-native：agent 产物是层层派生的（检索→综合→再加工）。审计、回滚、"这个结论基于哪些证据"都依赖派生图。原生 FS 只有 mtime，git 只有 commit→改了哪些文件，都答不出派生问题。ANFS 把它作为内核不变量。harness：provenance_benchmark.py
C4 · 兼容性 + 代价

agent 当普通目录用，零回归——保证不以兼容性为代价

真模型（deepseek-v4-flash）修 5 个 buggy Python 模块；每个修复在原生目录和 ANFS round-trip（物化→commit→重新物化到全新目录→pytest）两条路上各跑一遍。

原生目录
5/5 通过
ANFS 往返
5/5 通过

0 回归，integrity 全干净——在全新物化目录上跑测试通过，证明修复字节级挺过了 ANFS 往返。代价方面：ANFS 只对会话边界征税（materialize+commit+integrity：50 文件 66ms / 200 文件 278ms），不对每次编辑的内循环征税（worktree 就是普通目录，原生速度）。

为何是 agent-native：C1–C3 的保证若以"agent 用不顺手 / 成功率下降"为代价就毫无意义。C4 是让前三者可信的分母：agent 用 read/write/ls/grep 像用普通目录，guarantee 在边界廉价地兑现。这正是"是文件系统、不是数据库"的关键。harness：coding_agent_compat_benchmark.py · worktree_cost_benchmark.py

架构 · 小而硬的内核 + 投影

设计纪律（docs/06_kernel_contract.md / 09_complexity_audit.md）：内核只拥有那些无法由更简单事实重建的状态；其余一律是投影或适配器。新特性默认是投影，只有当它必须参与 replay、策略强制、冲突检测、持久溯源或完整性验证时，才进内核。

内核机制

内容寻址不可变

写入产生新 node，blob 按 SHA-256 去重。历史从不就地覆盖——这是溯源与完整性的根基。

有序事件日志

每次变更是一条 append-only 事件 + typed 输入/输出边，全局连续序列，支撑 replay 与派生图。

Copy-on-Write workspace

checkout/fork 投影 base ref，沙箱隔离，conflict-aware merge 在内核检测竞争。

集中策略 gate

所有 read/物化路径的字段/用途策略一律走 visibility.rs 的统一 gate，杜绝并行执行路径。

乐观并发

ref 版本 + IMMEDIATE 写事务 + busy 重试：竞争浮为干净冲突，而非静默丢失或裸锁错误。

可验证完整性

verify_integrity() 校验事件/ref/策略/merge/bundle 不变量；每个 conformance 场景以 == [] 收尾。

诚实的边界

机制，而非语义

ANFS 保证受标字节不过 API。它挡不住语义泄漏：secret 的改写、从非敏感字段的推断、模型跨轮记忆。C1 是字节级、提示词无关的保证，不是关于模型行为的断言。

其余范围

字段标签依赖保守 span 解析，无法识别的结构回退到整 node 保护（安全但更粗）；当前为本地单机内核，分布式协调是未来工作；C1/C2 机制 harness 用内置语料，PrivacyLens/SWE-bench-Live/LoCoMo 真实数据是发表前的下一步。

第五根支柱 C5（跨会话长期记忆）脚手架已就位（task_memory_benchmark.py：raw/extracted/hybrid recall、token-F1），接真模型 + 官方 LoCoMo 即可凑齐。
