底层所有 file 分三类

数据按"它在 agent 工作流里意味着什么"分类，而不是按字节。前两类是内核一等公民，对应 ref 的命名空间。

kind = "resource"

Resource

所有资源类原始数据：源代码、html、pdf、md、json、表格。任何 agent 需要读写处理的原始文件，也是唯一和业务/最终产出相关的部分。命名空间 resource:…

kind = "artifact" + event trace

Artifacts

agent 运行过程的中间产物与最终 trace。为溯源、为 debug、为优化 agent 而持久化。在 ANFS 里 = artifact ref + answer 输出 + 不可变事件日志本身。命名空间 artifact:…

设计位 · 非一等内核状态

agent 运行的赋能产物。有意不做进内核：可由 published artifact + 检索投影组合而成；memory 可看作 artifact 层与检索层的交互，留给上层自定义。诚实标注：当前非内核原语。

§01并发与一致性
今天的 FS

原生 FS 只知道 path/bytes/mtime/owner，不知道某文件属于哪个 run、对应哪个 patch 版本。多 agent 同时消费时出现 stale read / lost update / 跨文件状态不一致——而且不报错。

t0 Tester reads patch v1 → writes test_result(v1) = PASS
t1 Implementer overwrites patch → v2 (就地覆盖)
t2 Reviewer reads patch v2 + test_result(v1)
✗ 表面无错，语义已错：v2 从未被测过
我们的设计

不可变 node + 原子 publish：写入产生新 node，旧 v1 node 永不消失；ref 原子指向不可变 node。

ref 版本 MVCC + IMMEDIATE 写事务：并发改同一 ref 时乐观版本检查，败者拿冲突而非静默覆盖。

consume 只放行 published/approved，draft/stale 无法被下游读取。test_result 用 write(…, [v1_node]) 绑定它实际测的那个不可变 node——即便 v2 发布，血缘仍指向 v1，错配可被检出。

refs.rsworkspace.rsengine.rs::merge
怎么证明

8 个真进程并发 approve 同一 ref：

1 approved + 7 conflict，approve 事件恰 1 条，event_sequence 连续，integrity 干净。

真多进程 merge 竞争：恰 1 胜出 / 3 干净 RefConflictError / 0 静默丢失，败方数据可取回。consume 对 draft 直接拒绝。

§02血缘与溯源
今天的 FS

只记录 owner/mtime/size/permission 这类弱元数据，表达不了"patch.diff 是 Implementer 在 tool_call_17 基于 plan.md 和 issue.md 生成的"。这些只能事后从 trace 反推，而 trace 常不完整、且和文件本身割裂。

$ stat patch.diff
mtime=… size=… owner=agent_svc
✗ 不知道：谁生成、基于什么、哪个 tool call、哪个 run
我们的设计

血缘是内核原生能力，不是外挂日志。每条事件携带 agent_id / run_id / tool_call_id。

write(path, content, [source_nodes]) 在写时即记录 typed 派生边（source→result）。

查询：lineage_graph / lineage_nodes / node_events 给出 tool-call 级溯源闭包——价值不止 debug，还有 replay、权限传播、训练数据构造。

100%ANFS 派生闭包 4/4
git 0/4 · 0%——--follow 只给 report.md 自己的 commit 历史，结构上没有"派生自"关系。

§03Artifact 生命周期
今天的 FS

原生只有"文件存在/不存在"。但 agent 产物有 draft / partial / published / consumed / reviewed / approved / rejected / archived 等状态。

# 下游 agent 看到 patch.diff 存在
if exists("patch.diff"): consume()
✗ 它可能是半成品 / 旧版 / 未过 review
# review.md 存在 ≠ 已 approve
我们的设计

生命周期是一等内核语义。ref 状态机：draft → published → approved/rejected → archived，由 validate_state_transition 强制。

producer 在私有 workspace 生成 → 校验后原子 publish；下游 consume 只能拿 published/approved，draft 直接 PolicyDenied。

"文件出现了所以我就读"这种脆弱协作被从根上禁止。

refs.rs::stateconsume_implapprove/reject
怎么证明

14 个审批生命周期测试 + 状态机断言：

consume(draft) → 拒绝；非法状态转移 → 拒绝；只有 published 可被 approve；approved/published 才可被 consume。

每个生命周期场景以 integrity 干净收尾。

test_approval_lifecycle · test_posix_compat · test_conformance
§04访问控制与 blast radius
今天的 FS

ACL 在文件/目录/用户层，只回答"这个 user 能不能 open 这个 file"。但 agent 读多个文件、组合信息、派生产物再传下游——权限不能只在 open 时判断。

读 A.json ✓ 读 B.json ✓
✗ 但 A.name + B.salary 组合输出不该被允许
✗ Implementer 能伪造 test result
✗ Reviewer 能 approve 自己的 patch
# 一个 agent 被注入 → 污染整个共享空间
我们的设计

字段级策略标签：标签打在 JSON path / Markdown 字段的字节 span 上，由 visibility.rs 在查询/物化时统一门控（非文件级、非 open 时）。

用途门 + 能力规则：哪个 agent、什么 purpose、哪个 tool call 才能读/派生/发布。自我审批被拒。

派生数据策略传播：标签跟随派生过程，组合输出继承来源标签——权限随 artifact 流动，而非停在路径。

policy_labels.rsvisibility.rsspan_parser.rs
怎么证明

同模型/同防御提示/同攻击，只换上下文组装：

prompt 防御 75% 泄漏 · 正则 scrubber 25% · ANFS 0%

scrubber 抓住格式化 PII 却漏掉语义字段（工资 185000 / 病历号）；ANFS 受标字节从未进上下文。自我审批测试 → 拒绝 + 审计事件。

§05语义检索与索引
今天的 FS

只有 filename search / grep / regex。agent 要问的是语义问题："哪些文档和退款政策有关"、"哪些历史 run 解过类似 bug"——不一定含相同关键词，grep 易漏。外挂一个孤立向量库又会检索到旧版本、越权内容、过期 embedding，且无法追踪来源。

我们的设计

检索绑定到内核，不是外挂：FTS + node/chunk 向量都是可重建投影，键到具体 node id + 版本。

search/query 受策略门控：被标记节点不进结果（发现层也挡）。结果指向具体 resource/artifact id + 版本 + 内容切片。

索引一致性进 integrity 校验；rebuild_derived_indexes 可从 canonical 重建——FS 原生知道 embedding 是否过期。

query.rsembedding.rsintegrity.rs
怎么证明

策略激活后，受标节点在 search/query 的发现层命中数：

0越权命中（vs 无策略 100% 可见）

integrity 校验 node/chunk embedding 投影一致性；检索作为投影不进 canonical 真理源。

test_leak_redteam · test_search_answer · test_integrity

诚实：当前是 FTS + 本地向量，非托管语义索引。

§06可观测、Replay 与 Debug
今天的 FS

多 agent 失败时，最终输出不足以 debug——错误可能来自很早的某个中间 artifact。原生只存最终文件状态，答不出"哪个 agent 第几步读了什么、生成了什么、消费了哪个版本、哪条 policy 放行了"。

$ ls output/ final.md
✗ 错误最早出现在哪个 artifact？无从回到现场
我们的设计

file ops / publish / consume / handoff / policy decision 全进统一 append-only 事件日志，全局连续序列。

Time-travel：ref_view_at_event / materialize_ref_view_at_event 回到任意事件边界（比 commit 更细），重建当时的命名空间视图；replay checkpoint 证明回放边界。

events() 可按 run / tool_call / kind 过滤，支持复现整次失败 run。

events.rsreplay.rsobservability.rs
怎么证明

在 src2-publish 事件点重建视图：

只见 src1, src2，brief/report 尚不存在——精确到单次 mutation 的时间旅行。

report-publish 事件点 → 全部 5 个 artifact。两个边界均断言通过，integrity 干净。

§07隔离、沙箱与 workspace 分支
今天的 FS

agent 不该直接改共享主目录。裸沙箱/VFS 解决执行隔离，但只回答"agent 能看到哪些文件、是否污染 base"，不天然回答"这 artifact 发布了吗、基于哪些输入、被谁消费、release agent 能不能用"。

我们的设计

Copy-on-write workspace：checkout/fork_workspace 投影同一 base 快照，每个 agent 的改动写进自己的 delta ref——改坏可丢弃，多分支并行探索。

分支与 artifact publish / lineage / approval / merge policy 结合：私有分支试错 → 原子 publish → 下游消费不可变版本，而非随时被覆盖的 path。

= Git branch + DB transaction + workflow artifact store 的结合。

checkout.rsfork.rsconflict-aware merge
怎么证明

4 个进程各自 fork、并行改不相交文件、并发合并：

4/4 干净合并 · 0 误报冲突——证明不是靠"拒绝一切"取胜。

异行同文件场景三向对照：plain dir 静默覆盖 / git 静默自动合并 / ANFS 浮为冲突待审。

§08分布式与可移植
今天的 FS

真实 swarm 不在同一进程/机器：Planner 在一个 container、Coder 在 VM、Tester 在 K8s job、Reviewer 在另一 service。本地 /tmp 承载不了共享 workspace。NFS/对象存储能共享字节，但仍缺 agent 级版本/血缘/生命周期/权限/trace。

我们的设计

Bundle 可移植性原语：event / run / 全历史归档 bundle 的导出+导入，把一次 run 完整打包——resources 快照、artifacts、trace、lineage、policy decisions。

HMAC 签名做共享密钥溯源校验；replay checkpoint + archival_readiness 证明归档/回放边界完整后才允许物理压缩。

一次 run 可被打包、迁移、复现，而非散落的 tmp 目录。

bundle.rsreplay.rsgc_snapshot.rs
怎么证明

9 个 bundle 测试：事件/run/归档 bundle round-trip，导入侧重放并验证节点/事件/ref 一致，签名校验，归档完整性。

导入后 verify_integrity()==[] 且 ref 视图与源端一致。

test_bundles · bundle.rs

诚实：当前为单机内核，bundle 是可移植性原语；实时跨机协调是未来工作。

§09 · Kernel Discipline

小而硬：内核只拥有无法重建的真理

设计纪律（docs/06_kernel_contract.md）：新特性默认是投影，只有当它必须参与 replay、策略强制、冲突检测、持久溯源或完整性验证时才进内核。这条规则是上面所有保证能"硬"起来的原因。

内核 canonical 状态

  • 不可变内容寻址 blob + node
  • 可变语义 ref（带版本 + 生命周期）
  • append-only 事件日志 + typed 边
  • ref 生命周期审计行
  • workspace checkout base（CoW 边界）
  • 持久策略标签 + 字段级 fragment 标签
  • schema 迁移 + 完整性验证
blobs/nodesrefsevents/edgespolicycheckout bases

投影 / 适配器（可重建，非真理源）

  • search / query / FTS / embedding —— 检索投影
  • replay / 物化 —— 时间旅行投影
  • bundle import/export —— 可移植性投影
  • answer + token/cost —— 生成产物审计投影
  • worktree sync —— 普通文件工具兼容适配器

投影可被丢弃重建，因此可以快速演进而不威胁正确性。canonical 一旦写入即不可变。

兼容性是分母（C4）

以上保证若让 agent 用不顺手就毫无意义。真模型（deepseek-v4-flash）修 5 个 bug，在原生目录与 ANFS round-trip 两条路上：5/5 == 5/5，0 回归，integrity 干净。代价只在会话边界：

边界开销（每会话一次）

50 文件 = 66ms（materialize 28 · commit 29 · integrity 10）
200 文件 = 278ms

内循环（每次编辑）

ANFS 0.39ms ≈ 原生 0.23ms
worktree 就是普通目录，ANFS 不在路径里

§10 · Code-Graph Study

原生代码图：在公开多语言基准上对照编译器 oracle 验证

代码图是 ANFS 从不可变 node 派生出的原生投影（symbol=指向字节范围的 fragment，calls 边带调用点证据，call_graph/context_pack 落审计事件）。为验证它在真实大仓上成立，跑一个公开基准——7 个开源仓库、6 语言、每仓一道架构题（即 codegraph 用的那套，这里只当中立共享测试集借用）。把"哪些位置调用了符号 X"用 ANFS callers/call_graph（一次查询） 对比无工具 agent 的 grep，双方都对各语言编译器级 oracle 打分（rust-analyzer · scip-python · scip-typescript · scip-go · scip-java；Swift 无 SCIP 索引器，用 sourcekit-lsp 自建）。不挑题、不自己定对错。

诚实读法

ANFS 靠"读精确 span 而非整文件"省 token（干净完成处：gin 34%、okhttp 13%），但不稳赢——alamofire 反多 15%，且宽泛题常没答完（省的一部分是非完成假象）。而且 ANFS 调用次数更多（granular outline→read_span），与 codegraph"又省 token 又少调用"相反——我们今天是多调用换少 token。

真正稳的护城河

跨全 7 语言普适成立的是：精度更高 · 一次调用拿到带归属的检索 · 检索本身可审计/可回放（context_pack/call_graph 落 code_context_query/code_call_graph 事件）。后者是 grep 和只读 sidecar 都给不了的。token 省与完整性是语言相关的红利，不是保证。

§11 · Honest Boundaries

诚实的边界

机制，而非语义

ANFS 保证受标字节不过 API。它挡不住语义泄漏：secret 的改写、从非敏感字段的推断、模型跨轮记忆。C1 是字节级、提示词无关的保证，不是对模型行为的断言。

其余范围

字段标签依赖保守 span 解析，无法识别的结构回退到整 node 保护（安全但更粗）；Skills/Memory 暂非一等内核状态；分布式为 bundle 级而非实时；C5 长期记忆脚手架已就位，待接真实 LoCoMo + 模型。
